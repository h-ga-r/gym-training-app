# ジムトレーニング管理Webアプリ 仕様書（MVP）

## 概要・背景
学費のためジムを一時中断中。来年の再開に向けて、自分専用のトレーニング管理ツールを自作する。
HTML・CSS・JavaScriptのみで構築（バックエンドなし、localStorage永続化）。
使用端末：iPhone（Safari）。

## MVPスコープ
- インターバルタイマー
- トレーニング記録（種目・重量・回数・セット数）

以下は将来の拡張機能として今回は実装しない：
- 目標体型画像との比較・目標値算出
- 体重・体脂肪率をもとにしたPFC・カロリー自動計算

拡張しやすいよう、データ構造・画面には拡張の余地を残しておくこと（後述）。

## 1. トレーニング記録機能

### データ構造
```js
// 種目マスタ（部位ごとに分類、後から追加・非表示化が可能）
exerciseMaster: [
  { id: string, name: string, bodyPart: string, lastUsedAt: timestamp, isHidden: boolean }
]

// トレーニングログ（セット単位で1レコード。種目は exerciseId で参照。名前を後で変更してもログは壊れない）
workoutLogs: [
  {
    date: string,          // "2026-07-14" 形式
    exerciseId: string,
    setNumber: number,     // 1〜10
    weight: number,
    reps: number | null,   // 限界までの場合は null
    toFailure: boolean,
    recordedAt: timestamp
  }
]

// ユーザープロフィール（今回は空でよいが、後で目標体型・体脂肪率等を追加する前提でオブジェクトとして最初から分離しておく）
userProfile: {}
```

### 部位分類
プリセット固定リストを使用（自由入力にすると表記ゆれでグルーピングが破綻するため）：
`胸 / 背中 / 脚 / 肩 / 腕 / 腹 / 有酸素 / その他`

### 画面遷移フロー（カレンダー起点・セット単位記録）※変更後の全体フロー

1. 「記録する」ボタンをタップ → **当月のカレンダー画面**に遷移
2. カレンダー上で **今日の日付をタップ** → 部位選択画面へ
3. 部位ボタンをタップ（胸／背中／脚など）
4. その部位に属する種目一覧をボタン表示（＋一番下に「新規種目を追加」ボタン）
   - 新規追加時は部位は選択済みなので、種目名の入力のみでよい
5. 種目を選んだら「何セット目か」をドロップダウンで選択し、重量・回数を入力して1セット分を保存
6. 保存すると、**同じ日付・同じ種目の既存セット記録が画面上に一覧表示される**（例：1セット目終了後に2セット目を記録する際、1セット目の内容がそのまま見える）
7. 同じ種目で次のセットを記録する場合は、5〜6を繰り返す

### カレンダー画面の仕様
- 表示は当月がデフォルト
- 前月・次月に移動するボタンを設置する。ただし**当月より未来の月には進めない**（次月ボタンは当月表示時に無効化 or 非表示にする）
- その日にトレーニングを行った日付には、トレーニングした部位が分かるように色または文字で表示する（例：胸をトレーニングした日は「胸」のラベルやその部位に対応した色のドットを表示）
- 日付をタップすると、その日の記録内容（種目・セットごとの重量／回数）を確認できる
- **今日の日付のみ、タップすると記録画面（部位選択）に遷移する。今日以外の日付をタップした場合は閲覧専用とし、記録の追加・編集はできない**

### 表示順・削除仕様
- 部位内の種目は「最近使った順（lastUsedAt降順）」で表示する
- 種目を削除する際は物理削除ではなく `isHidden: true` にする（ドロップダウンには出さないが、過去ログの参照整合性は壊さない）
- 種目名の重複・表記ゆれチェックはMVPでは実装しない（後回し）

### 回数・セット目の入力方式（追加仕様）

**回数はドロップダウン選択式にする（自由入力ではない）**
選択肢: `3, 5, 6, 9, 10, 12, 15, 18, 20, 限界まで` の10択固定。
3・5の倍数以外の回数（例：8回）は記録できなくてもよい、という前提で確定。

「限界まで」を選んだ場合のデータ表現：
```js
{ weight: number, reps: number | null, toFailure: boolean }
```
`toFailure: true` の場合は `reps` は `null` とし、表示上は「限界まで」と出す。

**セット目はシンプルなドロップダウン**
1〜10の数値から選択（デフォルト上限10。それ以上のセット数を組む場合は後日拡張）。

## 2. インターバルタイマー機能

### 精度・実行方式
- `setTimeout` の相対時間だけに頼らない。開始時刻を `Date.now()` で記録し、常に「現在時刻 − 開始時刻」で残り時間を再計算する方式にする
- 開始時刻・設定秒数は毎回 localStorage にも保存し、JS変数だけに頼らない
  （iOSがバックグラウンド中にタブを強制リロードし、JS変数が消えるケースへの対策）

### バックグラウンド復帰時の挙動
- `visibilitychange` イベントで画面に戻ったことを検知
- 復帰時に localStorage の開始時刻・設定秒数から経過時間を再計算する
  - 終了していなければ、正しい残り時間からカウントダウン再開
  - すでに終了していれば、復帰した瞬間に点滅通知を開始する
- **既知の制約**：他アプリ閲覧中は画面描画・JS実行自体が止まるため、その間に休憩終了時刻を迎えても「その瞬間」には何も起きない。アプリ画面に戻って初めて気づく設計であることを前提とする

### 休憩時間の入力
- 固定プリセットボタン：60秒 / 90秒 / 120秒 / 180秒
- 各プリセットに `+10秒` `-10秒` の微調整ボタンを付ける

### Wake Lock（画面消灯防止）
- Screen Wake Lock API を使用し、タイマー画面をフォアグラウンドで見ている間は画面消灯を防止する
- 対応：iOS Safari 16.4以降（Ryoさんの端末なら問題なく利用可能）
- 制約：タブがバックグラウンドに回ると自動的に解除される。他アプリに切り替えている間の消灯は防げない

### 通知方式：画面点滅のみ（音・バイブレーションなし）
- iOS Safari は Vibration API 非対応のため、バイブレーション通知は実装しない（Androidであれば対応可能だが、今回は対象外）
- 音は周囲に迷惑になるため不採用
- 休憩終了時、背景色を反転・明滅させる視覚的な通知を行う
- **点滅は自動的には止まらず、画面をタップするまで継続する**（見逃し防止）
- **画面をタップすると、点滅が止まると同時に次のセット入力画面へ遷移する**（点滅停止と画面遷移を1つの操作にまとめる。これにより「タップするまで点滅し続ける」ことと「自動で次画面に進む」ことを両立させる）

### 音声自動再生ロック解除（将来Android対応・他機能拡張時のための布石）
- タイマー開始ボタンを押した瞬間に無音の音声を一度再生しておく処理を入れておく
  （iOS自体では使わないが、将来の拡張や他ブラウザ対応時に必要になるため仕込んでおくと安全）

## 既知の制約まとめ（iPhone / iOS Safari特有）
| 項目 | 制約 |
|---|---|
| Vibration API | 非対応。iOS Safariだけでなく、iPhone上のChrome/Firefoxも同様（WebKit強制のため） |
| Wake Lock API | iOS Safari 16.4+で対応。ただしバックグラウンドで自動失効 |
| バックグラウンド実行 | 他アプリ切り替え中はJS実行が完全停止。プッシュ通知なしでは「即時通知」は不可能 |
| データ永続化 | localStorageは端末・ブラウザ単位。同期されないため、必要であればJSONエクスポート/インポート機能を後日追加すべき |
